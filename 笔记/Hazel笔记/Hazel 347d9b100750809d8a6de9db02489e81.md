# Hazel

# 项目配置

链接过程选择动态库链接，不选择使用静态库链接是因为，使用静态库链接会把大量无用的代码编译进我们的游戏项目中

这样做大致有两点坏处

1:降低编译和链接速度，所有的代码都要在editor中编译一次再在game中编译一次，但是game中不需要编译所有的游戏引擎的代码，这样会造成性能浪费

2.不能控制哪些代码是引擎私有的，哪些是可以供外部访问的

所以综合考虑下来选择使用动态库链接，在引擎项目的属性中把配置类型设置为动态库即可

![image.png](image.png)

所有的配置都设置为64位版本，输出目录指的是最终生成文件的位置，中间目录则是过程中生成的obj文件的位置，把输出目录和中间目录设置为同一个文件夹下的两个文件夹，比默认的设置相比更加简洁，默认设置把所有的中间生成文件放在输出目录外，看起来特别乱

SandBox项目设置和上面基本相同，只是配置类型设置为exe

# 设置项目入口

无非就两种选择，把项目入口设置在SandBox项目中或者设置在Hazel中

如果把SandBox看做是我们的游戏项目的话，想象一下入口设置在这个项目下会出现什么问题。

游戏引擎中会有很多种系统，比如日志系统，内存系统等等，我们的游戏项目是在这个框架下运行的，需要有一系列基于引擎的初始化和基础设置

想象一下，如果入口点设置在游戏项目中，你每次运行游戏的时候都要写Hazel::Init()之类的初始化函数来保证引擎的各种基础功能正常运行，很麻烦，作为游戏开发者根本不需要在意这些不是吗

而且还有一点，假如我们的游戏同时支持安卓版本和window版本甚至支持更多种平台，每次在写游戏项目的脚本时都要写明项目入口应用的是哪个平台，如果有多个平台的话还要写多个入口，你每次写游戏的时候都要写一堆ifdef语句来判断当前游戏是哪个平台的，很麻烦不是吗

如果在引擎中设置入口点就可以避免所有的问题，打个比方就像是制定好了一个框架，所有的游戏项目就像这个框架中的一个小系统，你只要写好这个小系统就行了不用去在意自己的系统之外的东西。

既然已经明确了项目的入口点设置在了Hazel项目下，接着便是搭建我们的框架了

如果要引用动态库中的函数，好比你在动态库中写了一个void a()函数，你如果想在其它项目链接的时候能够找到这个函数，就要在这个函数前打上前缀

`__declspec(dllexport)`

而且在某个项目中应用的时候还要写上

`__declspec(dllimport) void a()` 

如果要链接的函数一多就会很麻烦不是吗，所以需要先解决这个问题

创建一个名为Core的头文件用于做一些预处理工作

```
#ifdef HZ_PLATFORM_WINDOWS
	#ifdef HZ_BUILD_DLL
		#define HAZEL_API __declspec(dllexport)
	#else
		#define HAZEL_API __declspec(dllimport)
	#endif
#else
	#error Hazel only supports Windows!
#endif
```

在该头文件中写下这一段代码，意思是如果定义了HZ_PLATFORM_WINDOWS，意思是如果该平台是WINDOWS系统，就进一步判断是否是需要动态链接的库(是否定义了HZ_BUILD_DLL)，如果需要动态链接，就将HAZEL_API定义为__declspec(dllexport)反之就定义为__declspec(dllimport)

这样，当需要标明某个函数是否需要动态链接时，只需要在前面打上标记HAZEL_API就行了

前提是已经定义了HZ_PLATFORM_WINDOWS和HZ_BUILD_DLL

# 日志系统

这里用到了spdlog库用来抽象我们自己的日志类，之所以还要封装自己的日志类，是因为在调用的时候，接口的形式当然是HZ_CORE_ERROR（）这种形式，而不是反反复复去使用spdlog库里的函数来输出日志信息

创建一个log.h和log.cpp文件，同时创建一个log类, 负责输出日志的平台有两个，一个是输出引擎内部的日志信息，比如引擎的内存系统，实体组件系统的日志信息，一个是用户在使用引擎时输出的日志信息。

日志系统全局存在且唯一，所以它是一个全局单例

给log类创建两个静态私有成员变量，一个作为引擎内部的logger，一个作为面向用户的logger

（静态成员变量需要在类内声明，在类外具体定义）

![image.png](image%209.png)

这里logger是一个共享指针类，这是因为创建logger的函数返回的是一个共享指针的类型，这样设计的原因是因为logger不像unique_ptr那样只能在特定的地方创建且在特定地方引用，但是logger需要在全局输出日志信息

之后由于需要调用s_CoreLogger→error(””)这样的函数，所以在类外需要获取我们的logger

故在public中创建两个函数

一个是getCoreLogger一个是getClientLogger，这两个函数分别返回对应logger的引用(使用引用避免了copy造成的性能开销)

![image.png](image%2010.png)

注意这两个函数也是静态的，因为只有静态函数才能调用静态成员变量

同样还需要有一个初始化函数，用来初始化logger的输出模式(同样是静态函数)

![image.png](image%2011.png)

在初始化函数中，set_pattern函数用于初始化logger的输出格式，stdout_color_mt函数则是实际创建了一个logger，好比s_CoreLogger = spdlog::stdout_color_mt(”HAZEL”)就是创建了一个名为HAZEL的logger并赋值给了s_CoreLogger

set_level(spdlog::level::trace)函数则是设置了日志输出的最低等级为trace

spdlog的日志库大概分为以下几个等级，由下到上等级一次增高

trace(追踪)

info(信息)

warn(警告)

error(错误)

fatal(致命)

这里把CoreLogger最低输出等级设置为了trace，意思是所有不低于trace 的日志信息均输出,而ClientLogger的最低输出等级设置为了error

这么设计是因为我们的引擎以后可能会有不同的版本，假如我们现在使用的是核心版本，以后发布时是发布版本，发布版本我们当然不输出所有的日志信息，一般只需要输出error和fatal就可以了

之后我们的日志系统基本就建成了，在入口点出需要先调用

HAZEL::Log::Init（）来初始化日志的各种设置(以后会用一个统一的初始化函数来初始化所有的设置)

之后再调用HAZEL::Log::getCoreLogger()→error()等函数来输出对应的日志信息了，但这样有些麻烦，每次想要输出日志信息的时候都需要调用前面那么长一串，为了简化，这里进行宏定义

![image.png](image%2012.png)

拿其中一个来说明这里是什么意思

`#define HZ_CORE_TRACE( ...)		::Hazel::Log::getCoreLogger()->trace(__VA_ARGS__);`

这里HZ_CORE_TRACE(…)中的…表示这个宏的参数接收任意数量的参数，__VA_ARGS__的作用就是把参数刚刚接收的参数全部移到这里，所以当我们调用HZ_CORE_TRACE(’’Ciallo～(∠・ω< )⌒★’’)的时候，就会弹出来对应等级的日志信息”Ciallo～(∠・ω< )⌒★”

# 事件系统

在游戏窗口运行中，可能发生各种各样的事件，比如鼠标点击，键盘按键等等，我们需要把这些事件中的数据传到某个函数中进行处理，好比你有一个控制角色移动的函数，我们就可以把键盘输入的方向信息传入这个函数中来控制角色的移动。

这里有一个总线的思想，windows中负责发送事件，把所有的事件发送到一个总线中，再通过该总线将事件信息分发到不同的层级中

先解释一些比较陌生的代码都表示什么意思

#### #define BIT(x) (1 << x)

![image.png](image%2022.png)

这里是定义了一个宏，1 << x 的意思是对1这个数字按二进制左移x位

好比1 << 2的结果就是(假设开始是0000 0001)0000 0100。

之后就可以设置一个枚举类用来存储所有的事件类别，这里没有用event class是因为之后要用到按位与运算，你也许会问为什么不用单纯的数字来表示各个事件的类别

这是因为有的事件可能同时属于两个类型，比如键盘输入事件就同时属于键盘事件和输入事件(EventCategoryKeyboard 和 EventCategoryInput)如果单纯使用整数，想要表示这种事件的话我就需要再定义一个EventCategoryKeyBoardInput，当事件的类型一多会变的非常麻烦，使用上述那种方式就可以避免这种情况，想要同时表示这两种事件，写下

EventCategoryKeyboard | EventCategoryInput就可以，| 表示或运算符，假如事件a用二进制表示为0000 0001 事件b用二进制表示为0000 0010，经过或运算符之后a | b的值就为0000 0011，之后如果想判断这个事件是不是属于某个类别

好比我把event记为EventCategoryKeyboard | EventCategoryInput,我只需要通过event & EventCategoryKeyboard就可以判断这个事件是不是键盘事件，&是与运算符，它的逻辑是这样的，同样假设a | b 事件为0000 0011记为event1，如果要判断该事件是否同时是a事件0000 0001，

只需要event1 & a就可以

此处的结果为0000 0001，值不为0(true)就表示该事件同时是a事件，假如我有一个c事件0100 1000，经过与运算后event1 | c的结果为0000 0000的值为（0）表示该事件不是c事件

这里就解释了为什么不用enum class，因为使用了这个之后，enum里的成员就不会被当做整数处理，而我们需要把他们当做整数处理来做位和与运算

#### using EvenFn = std::function<bool(T&)>;

![image.png](image%2023.png)

这行代码类似于#define ，就是把所有的EvenFn当做std::function<bool(T&)>,好比EvenFn<int>的意思就是std::function<bool(int&)>,使代码可读性更高

#### ##运算符

把参数中的内容强制粘贴，好比#define A(string) “a”##string,它就把”a”和string强制粘在了一起，结果就和” a “ + “string“结果相同，但是这里有一个要求，经过粘贴后的结果是一个单一变量，好比变量a和变量b粘贴，结果ab必须是一个存在的单一变量，cherno的那种写法EventType::##type在比较新的visual studio中是会出错的

#### #运算符

把输入的参数当做字符串处理好比#define(A) #A

## event基类

![image.png](image%2024.png)

在实际的运行中某个事件我们不用重复处理，好比鼠标点击一次你只需要处理一次，用bool来表示该事件是否已经处理

为了方便调试，我们需要在所有的event中查看当前的event信息，一个event所包含的信息除了是否被处理(m_Handled),还要有这个event的名称，category信息，以及个别event的一些数据(比如MouseMoved中的鼠标坐标)，

GetEvenType函数获取这个event的枚举值，

GetName函数用来获取该event的名称，

GetCategoryFlags用来获取该Event的Category标签，

ToString用来获取某些特定事件的数据，

IsInCategory用来判断该事件是否属于某个类别。

除了IsInCategory之外，其它函数都需要被子类覆写，所以标记为虚函数，又因为都是getter，所以在函数后打上const，=0表示这个函数为纯虚函数，必须要被覆写，(表示不改变成员变量，代码可读性更高)

GetEventType,GetName,Tostring函数对于大多数事件来说，函数功能的重复性是比较高的，可以通过写一个宏来处理，需要特别处理的事件再单独覆写，能够省去很多代码，并且可读性也更高

## EventDispatcher

![image.png](image%2027.png)

这个类就是用来处理回调事件的核心，它有一个私有成员变量为m_Event，这里使用引用是为了避免copy，将所有对m_Event的修改都作用在变量本体上，它的作用就是使特定的dispatcher对象调用作用域特定类型事件的函数。

构造函数中初始化m_Event即可

核心函数是bool Dispatch(EventFn<T> func)，直接讲可能比较抽象，这里举个例子会容易的多。

假如你某一个集中处理回调事件的文件中，创建了一个dispatcher(MouseMoved)，它的私有成员初始化为了这个MouseMoved事件，之后你想要调用这个事件中的某个函数，你只需要使用

dispatcher.Dispatch<MouseMoved>(func)来调用这个func函数就可以了，这里需要做一个类型判断，如果这个函数的类别和这个dispatcher私有成员的类别相同，则可以调用这个函数，反之则不行

如果可以调用则返回m_Event.m_Handled = func(* (T*))&m_Event)，这里既调用了func函数，又标记了m_Handled是否被处理，同时又返回值判断这个dispatcher是否能调用这个函数，可谓是一石三鸟。

*(T*)的意思是讲&m_Event强制转换为T类型的指针之后再解引用,你也许会问，这里不是已经确定m_Event是T类型了吗，为什么还要强制转换一下，这是因为编译器并不知道m_Event是T类型，这里只是cpu经过了判断说明m_Event为T类型，区别就是cpu判断是但代码运行到这里时才做的判断，而编译器则是在运行的那一刻就确定好的，在编译的时候还没有把m_Event的类型确认

如果直接写return m_Event.m_Handled = func(&m_Event)这里就会报错，编译器知道func函数是作用于T类型的变量，但它不知道m_Event到底是属于什么类型

有了这个函数之后，在做事件处理时，就不用写一堆if else语句了，就像这样

void OnEvent(Event& e) {
if (e.GetEventType() == EventType::KeyPressed) {
KeyPressedEvent& kpe = (KeyPressedEvent&)e;
OnKeyPressed(kpe);
} else if (e.GetEventType() == EventType::WindowResize) {
WindowResizeEvent& wre = (WindowResizeEvent&)e;
OnWindowResize(wre);
}
// ... 无限循环

而且由于EventType是一个整数，这里的判断速度非常快

这里就可以看出上面的static函数的作用了，可以通过T来直接获取当前的EventType枚举值做判断

在明确了Event基类和EventDispatcher类的作用之后，event子类的覆写就很简单了

这里只举一个例子就行，其它的都是比葫芦画瓢

## 例:KeyEvent

![image.png](image%2028.png)

所有的键盘输入事件同时为Input类型和Keyboard类型，并且都需要一个私有成员变量来存储键盘按键信息，所以这里创建一个基类KeyEvent继承自Event，KeyPressedEvent和KeyRealeasedEvent都继承自该类，构造函数用来初始化p_KeyCode（KeyCode按键信息需要传递给子类，又不希望外界修改，所以设置为protected），同时给所有键盘事件打上类别标签

`EventCategoryInput | EventCategoryKeyboard`

因为键盘事件有的时候需要判断是不是长按，所以KeyPressedEvent私有成员中记录了repeatCount，如果repeatCount为0就说明只按了一下，反之为长按

设置一个Getter获取repeatCount以便偶尔调用，同时覆写ToString函数用来输出键盘输入事件的具体信息(具体是哪个按键和重复次数)
